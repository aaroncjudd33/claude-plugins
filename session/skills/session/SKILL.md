# Session Skill

Governs session lifecycle across all project types (plugin, story, cab, personal, general).

## Terminology тАФ the canonical glossary (acp-ajudd#64, #70)

**This section is the single source of truth for what each load-bearing term MEANS.** Every other doc тАФ `references/*.md`, `commands/*.md`, the Confluence page тАФ may describe how a term *behaves in a flow* (its mechanics) but must **not re-define the term**: it points here instead. The payoff is bounded but real (acp-ajudd#70): a **conceptual** change (what a term means, how the roles relate) becomes a one-entry edit here instead of an N-file rewrite. A pure **rename** (e.g. `work` тЖТ something else) still needs a find/replace of the term wherever it is *used* тАФ the glossary removes the N re-explanations, not every occurrence.

### The three "session"-family words

Three words, kept distinct so the word "session" never has to carry two meanings at once:

- **terminal** тАФ a Claude Code conversation (the CLI window / instance you are typing into). Claude Code itself calls this a "session"; this skill deliberately says **terminal** to remove that collision. Several terminals can run on one machine at once and collaborate through the inbox and copy-paste handoffs (┬з The three roles).
- **coding session** тАФ the **file-backed** work unit this plugin tracks: one `<name>.md` state file under `session_root`, born only by `code`-ing work. **Always call the file-backed unit a "coding session," never a bare "session"** тАФ planning / refine / dispatch are **sessionless** roles (no file). This is "the file dictates the mode": a target that has a session file is a coding session; a target with none is planning.
- **session** тАФ the umbrella term and the plugin / command name (`/session:*`). It names the whole working system тАФ plan, dispatch, code тАФ which is exactly why the plugin keeps the broad name (renaming it to something narrower like "impl" would mis-fit and carry a huge blast radius). In prose, prefer the qualified **coding session** whenever you mean the file-backed thing.

**Vocabulary only тАФ nothing is renamed. `/session:*` command names and the plugin name are unchanged (acp-ajudd#64).**

### Glossary тАФ one line each; the pointer names the section that owns the term's mechanics

- **terminal** тАФ a Claude Code conversation; several can run and collaborate on one machine. тЖТ *this section, above; collaboration in* ┬з The three roles.
- **coding session** тАФ the file-backed (`<name>.md`) work unit, born only by `code`-ing work; the sole thing this plugin persists. тЖТ ┬з Session Types, ┬з The file dictates the mode.
- **session** тАФ umbrella term for the whole system and the plugin / command name; qualify as **coding session** when you mean the file. тЖТ *this section, above.*
- **stance** тАФ the behavioral posture of a context, one of **planning** (default, sessionless) or **coding** (explicit, immutable, backed by a session file); read from file-presence, never a stored flag. тЖТ ┬з Session Stance.
- **the three roles** тАФ **refine**, **dispatch**, **code** тАФ the operational unit of the dispatch model; each context knows and declares its own role. тЖТ ┬з The three roles.
  - **refine / refinement** тАФ the **planning role that authors the inbox**: scopes `work`, mints IDs, promotes `capture`тЖТ`work`; sessionless; the one inbox writer. Entered by the `refine` verb. тЖТ ┬з The three roles, `commands/refine.md`.
  - **dispatch** тАФ the **planning role that coordinates**: reads/bundles/sequences `ready` `work`, hands notes to `code`, validates the returned tree, decides **validated / safe-to-close**; sessionless, read-only + notes-only тАФ with one named exception: may author clear, no-refinement **splits** of in-flight work (acp-ajudd#95), never requirements. тЖТ ┬з The three roles, ┬з Inbox write-authority, `commands/dispatch.md`.
  - **code / coding** тАФ the **coding role that implements**: folds in the `ready` `work` and builds; fresh per build; the only role with a session file. Entered by the `code` verb. тЖТ ┬з The three roles, ┬з Session Types.
- **capture (the role)** тАФ the sessionless **ideation station *upstream* of the three roles** (the head of the pipe `capture тФАтЦ╢ refine тФАтЦ╢ dispatch тФАтЦ╢ code`): banks a raw idea, optionally peeks for a **one-line viability sniff**, and writes a `capture`-type entry (or drops it) тАФ never scopes, designs, or codes. Distinct from **capture (the type)** below, which is the entry it produces. Entered by the `capture` verb / `/session:capture`; plugin/personal only. тЖТ ┬з The three roles (the ideation station, acp-ajudd#96), `commands/capture.md`.
- **work** тАФ a thing to build (the generic buildable unit; a Jira story is work's work-repo form, an `_inbox/` `work` entry its plugin/personal form). тЖТ `references/inbox-convention.md` ┬з Inbox Model, ┬з State-Exclusivity.
- **capture (the type)** тАФ inbound info handed to a session (a note / data payload / stray idea); read-and-dispositioned, not scoped, unless promoted to `work`. The output the **capture role** (above) produces, and what a `/session:inbox` capture drops. тЖТ ┬з Captures Inbound, `references/inbox-convention.md` ┬з Captures inbound.
- **maturity lifecycle** тАФ the `new` тЖТ `refining` тЖТ `ready` stages a `work` entry passes through (rough тЖТ iterate тЖТ ready; Heber's Jira flow); orthogonal to pickup state. тЖТ `references/inbox-convention.md` ┬з Inbox Model, `commands/refine.md`.
- **handoff note / handoff block** тАФ the self-contained, role-aware paste block that carries the **run** (what to do, watch-fors, report-back) between terminals; distinct from the `work` entry, which carries the **spec**. тЖТ ┬з Cross-Session Paste Handoff.
- **capture disposition** тАФ the fate applied to a `capture` on read: **promote** (тЖТ `work`), **discard**, **absorb**, or **feed a refinement**; a non-completion outcome stamped `[DISPOSITIONED тАж тАФ <fate>]`. тЖТ ┬з Captures Inbound, `references/inbox-convention.md` ┬з Disposition & completion.
- **depends-on** тАФ an optional `> [depends-on: <id> тАФ <reason>]` marker declaring a sequencing prerequisite; `refine` writes it, `dispatch` honors it (an unmet dependency = not dispatchable). тЖТ ┬з The three roles (prereq-check), `references/inbox-convention.md` ┬з Sequencing.
- **HALT / halted** тАФ the clean mid-flight stand-down of dispatched work (`Action: HALT` out / `State: HALTED` back): no publish, no commit, WIP preserved; re-enters as a NEW entry citing the halted one. тЖТ ┬з HALT.
- **consumed = frozen** тАФ the state of a `work` entry after `code` picks it up (fold-then-archive): it takes **no further writes from any role**; changes become a new entry. тЖТ ┬з State-Exclusivity, ┬з The three roles ┬з Inbox write-authority.

**Refine, in one line and pointing onward:** the **analyze-then-scope** planning flow that scopes raw requirements into `work` before any code is written тАФ sessionless in every zone (the work it produces is itself the WIP store *and* the deliverable), targeting **strictly by zone** (plugin/personal тЖТ a `work` entry; work repo тЖТ a Jira story; general тЖТ nothing), and it **never offers to `code`** (acp-ajudd#56). Full mechanics: ┬з Session Types, ┬з Session Stance, and `commands/refine.md`. Legacy `refinement-*.md` files from the old model are harmless тАФ hidden from the default listing, skipped by `session:migrate` тАФ and can be deleted whenever convenient.

## Session Types тАФ the organizing principle

Sessions are typed, and the type determines whether there's an **external system of record** тАФ which in turn determines how sessions are created. This is the distinction that drives everything below.

| Type | System of record | Created how | Session-command enforcement |
|------|------------------|-------------|------------------------|
| **plugin** | none тАФ the session *is* the work | item-driven only: `code` a refined `work` entry (new work is `refine`d first); named after the **feature** | command-level: session commands require a session |
| **personal** | none тАФ the session *is* the work | item-driven only (identical to plugin) | command-level: session commands require a session |
| **story** | Jira story (BPT2-XXXX) | keyed to the story; `code BPT2-XXXX` | command-level: session commands require a session |
| **cab** | CAB card (Jira) | keyed to the CAB; `code cab <keys>` | command-level: session commands require a session |
| **general** | none (lightweight) | `code <name>` (named by the user) | command-level: session commands require a session |

**Why plugin/personal differ from story/cab:** story and CAB work already have an authoritative external unit of work (the ticket) that says what's being done and tracks its lifecycle. Plugin and personal work have no such anchor тАФ so the session itself becomes the unit of work: it can only exist by `code`-ing (graduating) a refined `work` entry, and it's named after the feature. That is what makes "the session is the work" real rather than aspirational. (The *creation path* differs by type; the *command-level enforcement* тАФ session commands need a session тАФ is uniform across all types.)

**A session file exists only for implementation тАФ it is always a coding session (1:1).** Planning and refinement are **sessionless**: they are the `refine` flow, which writes **work** (a Jira story in a work repo, or a `work` entry at `refining`/`ready` in plugin/personal) and never a session file. There is no session "mode" тАФ **the file dictates the mode** (see below): scoping happens in `refine` (no file), building happens in a session (one file). `ready` work is graduated into a coding session by a separate `code` gesture; refine itself never crosses that line.

### The file dictates the mode тАФ two verbs (acp-ajudd#56)

**You are never in a "mode" you set тАФ the mode is read from the file you're touching.** A target with **no session file** is a *`work` entry* (or Jira story) тЖТ **planning / refining**. A target that **has a session file** is *coding*. The verb a user types just names which side they're on, so the vocabulary and the file state are the same fact said twice. There are exactly **two verbs**, and `new` / `resume` / `pick` are all retired into them:

- **`refine [target]`** тАФ **planning, sessionless.** Edits **work** (a `work` entry in plugin/personal; a Jira story in work repos) and **never creates a session file.** Bare `refine` = new work from scratch (this absorbs the old `new` тАФ new work begins by scoping it); `refine <n|id|KEY>` = resume editing existing work in place. Refine is also where a `capture` gets **promoted to `work`**.
- **`code <n|name|KEY>`** тАФ **coding.** Be in a coding session. If the target is a **`work` entry** тЖТ graduate it and a session file is born (Jira story тЖТ *In Progress*; `work` entry тЖТ consume/fold-archive). If the target **already has a session** тЖТ resume it (this absorbs the old `resume`, and graduating work absorbs the old `pick`). One verb; the file decides which.

`new` reintroduced coding-without-a-scoped-unit, so it's gone тАФ scope-first *by workflow*, not by policing (acp-ajudd#1). `resume` was redundant with `code` тАФ `code BPT2-6508` and the old `resume BPT2-6508` did the identical thing, since the file already tells the system whether to graduate or re-enter. **Cross-zone uniformity is the payoff:** identical two-verb feel everywhere; only where the *work* lives differs (a `work` entry vs a Jira story). A dev in virtual-office types `code BPT2-XXXX`; a plugin session types `code <n>`. Same word, same meaning. **Warn, never block** on `code` of ungraduated work (acp-ajudd#1); **refine never offers to `code`** (see ┬з Session Stance and `commands/refine.md`).

## Session Stance тАФ planning is default, coding is explicit and immutable (acp-ajudd#28/#30/#32)

A working context is always in one of two **stances**. The stance is a *behavioral* posture, **not** a stored flag тАФ there is no `Mode:` field (acp-ajudd#16's storage call stands: a session file = coding, 1:1). It is decoupled from file-presence in the sense that *behavior* is what defines it, but the two line up exactly: **planning is sessionless; coding is backed by a session file.**

**The operational tell (answer "which stance am I in?" from one fact).** You are in **coding** stance *iff a session file exists for this context* тАФ this is the same "the file dictates the mode" fact stated as a stance. No session file тЖТ **planning** (reading, searching, discussing, and refining a `work` entry are *all* planning). `code`-ing a target (graduating `ready` work, or resuming a session) is the **sole** action that mints/opens the coding session file and puts you in coding; from that point the stance is immutable (see the keystone rule below). **A `work` entry is NOT a session file** тАФ reading or refining it is still planning; `code`-ing it is the gesture that creates the file. So **the command you reach for is itself the stance declaration**: `refine` is the planning verb (writes work, no file), `code` is the coding verb (mints/opens the session file).

**Planning is the default stance.** In it, read, search, analyze, discuss, and **write work and captures** (inbox entries, Jira stories) freely тАФ but make **no code changes**, and **never self-start a coding session**. Planning stays sessionless; its output is work and captures, not commits. (This restores an *explicit* default: acp-ajudd#16 correctly removed the `Mode:` field, but defining planning as merely "the absence of a session file" left the stance implicit and driftable тАФ which is how a planning conversation once self-started a coding session. The stance is the default; the file is just its footprint.)

**Coding is entered only by an explicit user gesture** тАФ the user `code`s `ready` work, says "switch to coding" / "build it", or runs `/session:start code <target>`. Claude may *offer* ("hand this to a `code` session?") but **never crosses on inference** тАФ and **refine specifically never offers to `code`** (acp-ajudd#56): graduation is always the user's own separate `code` gesture. Capability containment still holds тАФ coding тКЗ planning (a coding session may plan within itself; planning may not code) тАФ but in practice a coding session rarely plans-then-builds; it folds in the `ready` work and moves.

**A stance is immutable once established (acp-ajudd#32 тАФ the keystone).** Once a context is planning it stays planning; once it is coding (has a session file) it stays coding. **There is no "switch type" verb.** The explicit gesture that enters coding starts a **fresh** coding session тАФ it **never converts the planning context in place.** The only planningтЖТcoding path is **handoff into a fresh session** (`/session:handoff` from planning, then in a fresh context the receiver's `code` gesture тАФ `/session:restore`, or `/session:start code <target>` тАФ opens the coding session). In two-verb terms: planning hands off *to a `code` session*, and the coding session hands the verified result *back to `refine`/planning*.
- **Why тАФ the boundary IS the value.** The clean, auditable line between "what was decided" (planning) and "what was built" (coding) is exactly what the handoff/spawn/inbox machinery exists to carry. In-place conversion destroys it two ways: (a) the planning context's exploratory reasoning, rejected approaches, cross-item chatter, and ID-minting would **bleed into the implementation** тАФ the scope-bleed we prevent; a fresh coding session starts from *only* the distilled handoff (a feature, not a loss). (b) In-place is always lower-friction than writing a handoff, so if allowed it gets used, the handoff **atrophies**, and the decidedтЖТbuilt lineage is lost. **Locking the stance is what forces the handoff to exist.**
- **Symmetry.** codingтЖТplanning is *already* locked by fileless-ness (planning has no file; you can't drift backward without abandoning the coding file). Making planningтЖТcoding equally immutable is simply the symmetric, cleaner rule.
- **One-directional in practice тАФ no ceremony on coding sessions (Aaron).** The lock's real weight is entirely on planningтЖТcoding. The containment path (a coding session planning within itself) carries little weight and needs **no guarding** тАФ do **not** add ceremony to coding sessions to honor this.
- **The cheap exit (acp-ajudd#29).** `/session:handoff` works from a sessionless planning context and writes a **durable resume file** a fresh session picks up, so crossing the boundary costs one command and loses nothing. Without it the lock is friction; with it, immutability is nearly free and strictly cleaner.

**One inbox writer + one coding session per repo (acp-ajudd#30, refined by #57).** A documented discipline (instruction-level, **not a hook**), prompted by a live incident: two concurrent planning contexts on one slug raced the shared ID counter and nearly minted a duplicate. The cap's **real meaning is "one *writer* of inbox entries," not "one sessionless session"** тАФ a distinction the dispatch model (┬з The three roles) makes load-bearing:
- **Cap coding at 1:** `_active`, `_index.md`, and the git working tree / push are **coding-only writes** тЖТ one coding session = one writer, so structural races (a clobbered `_active`, a lost `_index` row, cross-contaminated commits, push races) cannot happen.
- **Cap the requirements writer at 1 тАФ that writer is `refine`.** Defining/refining requirements and writing them into the `_inbox/` dir is the `refine` role's job; two requirements writers race the counter (the incident). **A second sessionless context is legal when it is not a *requirements* writer** тАФ that is exactly what the `dispatch` role is (a reader/coordinator that authors only clear, no-refinement splits of in-flight work тАФ acp-ajudd#95 тАФ every write minted through the atomic `inbox-id.py`, so it never races the counter or forks a live entry; ┬з The three roles). So "one planning session" is more precisely **one requirements author (refine) plus coordinators (dispatch) that write only no-refinement splits**.
- **Residual:** the counter race itself is closed **in code** by acp-ajudd#31 (atomic increment in `inbox-id.py`) and acp-ajudd#66 (the mint now self-heals тАФ it seeds from `max(stored-counter, highest #N across the slug's `_inbox*.md` files) + 1`, so a lagging counter or a hand-written header can never yield a duplicate ID; hand-writing headers is banned тАФ always mint via the script). **Immutable stances (acp-ajudd#32) keep the counts legible** тАФ a session never morphs from one type into the other, so "one writer of each kind" stays a countable invariant rather than a moving target.

This is the behavioral layer above ┬з Session Enforcement (which enforces only "engage a session command тЖТ a session must exist"). Stance is never policed by a hook тАФ like everything here, editing is never blocked (acp-ajudd#1); the stance is a convention that keeps the planning/coding boundary clean.

## Session Enforcement (command-level тАФ acp-ajudd#1)

**Enforcement lives at the command level, not the file-edit level. Editing is never policed.** The plugins exist to *record work in sessions* тАФ but you cannot actually stop anyone from editing code (anyone can open a plain Claude Code terminal in any repo and do whatever, and that is fine). So the thing worth enforcing is not "block edits"; it is: **if you engage the session workflow, a session has to exist.** That is enforced by the session commands themselves.

**The rule:** any command that operates on "the active session" fails gracefully at the top if none exists for the current slug тАФ a clean stop with the fix, never a crash:

```
No session established for <slug>. Run /session:start first.
```

- **Enforced (require a session):** `commit`, `finish`, `checkpoint`, `switch`, `spawn`, `store`, `restore`.
- **Exempt (run without a session):** `start` (it creates one), `refine` (sessionless тАФ writes directly into the `work`), and the read-only views `search` / `worklog` / `in-flight` (they just report "no active session" where relevant).

**Sanctioned wording exception тАФ `restore` (acp-ajudd#39, C9).** Six of the enforced commands key off `_active` and emit the exact clean-stop string above. `restore` legitimately differs: it keys off the `_context_*.md` files (not `_active`), so its clean stop reads `No stored context for <slug>. Run /session:store before /clear, or /session:start to pick up work.` This divergence is intentional and correct тАФ the message names the artifact `restore` actually looks for тАФ not drift to be normalized to the standard string.

**No edit-blocking hook.** There is no `PreToolUse` hook on `Edit`/`Write` тАФ the old `session-scope-guard.py` was deleted (acp-ajudd#1). Editing files, in any zone, is never blocked by the session plugin. `sessionGate.enforce` is retired along with it; a fresh install never blocks edits anywhere.

**Reads, searches, and read-only commands are never gated тАФ investigation is always free, for every type.** (This was true before and stays true.)

The `session-file-guard.py` (Read-hook injection scan) and `session-commit-guard.py` (git pre-commit PII/secrets guard) are unrelated to workflow enforcement and remain in place тАФ they protect file *content*, not the *workflow*. This command-level model is what the personal global CLAUDE.md "Session Enforcement" instruction now describes.

## Development Lifecycle тАФ Polymorphic Commands

`start` / `commit` / `finish` are the three stages of the development lifecycle, and they behave **polymorphically** across every environment: **start = pick something up ┬╖ commit = iterate on it ┬╖ finish = we're done.** The *shape* is identical in every zone; the *sources and functions* differ by session type. Same command, environment-appropriate behavior тАФ it just works the same regardless of which project you're in.

| Stage | plugin | story / cab | personal / general |
|-------|--------|-------------|--------------------|
| **start** (pick up) | `code` a refined inbox `work` entry тЖТ feature session (new work `refine`d first) | Jira story/CAB kickoff via `code` | `code` a `work` entry (personal) or `code <name>` (general) |
| **commit** (iterate) | **local commit only тАФ never push** | commit **and push** + Jira comment | commit and push |
| **finish** (done) | **the deploy** тАФ bump version тЖТ push master тЖТ reinstall (live) | Jira close, CAB handoff, Confluence, Teams | close out; no deploy concept |

**The plugin commit/finish split (why it exists):** for plugins, `master` **is** the deployed branch тАФ reinstall pulls straight from it. So a mid-session `commit` must stay **local** (a private safety checkpoint); pushing unversioned WIP would let the marketplace pull half-finished work. The version bump + push + reinstall is a **deploy**, and a plugin deploys only when the work is *done* тАФ as its terminal-content action (after all state is settled тАФ code lands last). This mirrors how story/cab already separate iterate (commit тЖТ push/Jira) from done (finish тЖТ CAB handoff). `commit` **never** bumps the version or reinstalls for any type.

**Ship тЙа close тАФ the table's "finish = the deploy" is refined by acp-ajudd#94.** The deploy (ship) and the record-close are un-bundled: on the orchestrated dispatchтЖФcode happy path the coding session **ships at build-end on its own authority** and stays `active`, and the explicit **close** (`/session:finish`) runs later on the validated signal (┬з The dispatchтЖФcode loop). So `finish` no longer *is* the deploy in every case тАФ it is the **close**, which *deploys-if-not-already* (the deploy for a solo session that never separately shipped; a no-op for an orchestrated one). `commit` is unchanged (still local-only, never a deploy).

## Loaded Memories & Commit Tracking

Two session-file fields connect the session lifecycle to the **memory plugin** and to git history. The session plugin owns the schema (writes them in checkpoint/finish/commit/start); their content is produced elsewhere.

**`Loaded memories:`** тАФ feature memories pulled into context during the session, recorded by the memory plugin (`/memory:load`, `/memory:scan`, `/memory:save`). Format:
```
- **Loaded memories:**
  - checkout-payment-validation  [feature:checkout/payment-validation]
```
- **Written by:** the memory plugin. **Preserved by:** checkpoint/commit (carry forward unchanged). **Surfaced by:** start (resume block, with a `reload` option). **Validated by:** finish (the memory-validation batch item runs `/memory:groom` against this list; deleted memories drop out).
- This is the anti-rot loop: a memory only lands here if it was loaded for real work, and everything here gets an accuracy check at finish. Memories never loaded stay inert.

**`Commits:`** тАФ running list of commits made during the session, written by `session:commit`. Format:
```
- **Commits:**
  - [2026-06-12] a1b2c3d тАФ fix: validation error on empty cart  https://github.com/org/repo/commit/a1b2c3d
```
- **Written by:** session:commit (appends the short SHA + subject; **GitHub link for non-plugin types only** тАФ plugin commits are local/unpushed, so a commit URL would 404 until the finish deploy pushes it). The plugin **finish deploy (Step 11)** also appends its deploy commit here. **Preserved by:** checkpoint/finish/start. **Surfaced by:** start (resume block shows the most recent 3).
- Joins the other session artifacts (story, CAB, PR, deploy links) so returning to a session surfaces everything that was produced in it.

Both fields are **omitted entirely** when empty тАФ older session files without them load fine (backward compatible). The memory plugin reads/writes `Loaded memories:` through this skill's Path Resolution to find the active session.

## State-Exclusivity тАФ a live `work` entry OR a consumed session, never both (acp-ajudd#13)

A given piece of work is **either a live `work` entry or a consumed coding session тАФ never both at once.** This **replaces the old "coding must not edit the work" role rule**: rather than forbidding a coding session from touching requirements, the model makes divergence structurally impossible. It is a **documented convention, instruction-only тАФ no guard or hook** (consistent with acp-ajudd#1's "editing is never policed"; a work-layer hook would re-police the memory tier we keep free and would be trivially bypassed anyway).

- **A coding session *may* edit a live `work` entry.** While it is still in the inbox, editing its body is just planning-in-the-moment тАФ free-rein, like `refine`. There is no prohibition on a coding session shaping requirements.
- **Picking up a `work` entry consumes it тАФ fold-then-archive (acp-ajudd#40).** The pickup folds the body into the session file and **removes** it from the live inbox, appending a `[CONSUMED <date> тЖТ session <name>]` copy to `_inbox_archive.md` first as a recovery net (stable `<id>` preserved in the session's provenance). Afterward there is no *live* entry left to edit тАФ the work is session-only, so it can never exist as *both* a divergent live entry and an in-flight session. The archived copy is history (not a second live record) and the `<id>` is retired, never reused. **Self-enforcing**, not policed.
- **Consumed = frozen тАФ no post-conversion writes, changes spawn new work (acp-ajudd#59).** The corollary of consumption: once converted, the entry takes **no further writes from any role** тАФ not refine, not dispatch, not the coding session itself. Anything the build turns out to need becomes a **new `work` entry / story**, never a reopen of the consumed one (implementation moves fast; nothing waits on the retired entry). This is a hard line, and it is the inbox half of the per-role write-authority set in ┬з The three roles ┬з Inbox write-authority. (Distinct from *halted* work тАФ stood down **before** completion тАФ which likewise never reopens but re-enters by **citation** in a new entry; see ┬з HALT.)
- **Jira stories keep the "locked once *In Progress*" rule.** A story isn't consumable (can't be fold-then-archived), so exclusivity can't be structural for it; `/story:update` locks its description at *In Progress* to get the same "requirements don't drift mid-build" outcome. Exclusivity-by-consumption is **inbox-native only**.

A coding session still freely writes its **own session file**, and a `refining`тЖТ`ready` **status flip** plus the fold-then-archive on pickup are normal lifecycle, not forking. It may also **post NEW inbox entries** (handoffs via `/session:inbox`, spawns, captures) тАФ those create fresh entries, they don't fork an in-flight one тАФ **but only when it is NOT part of an active dispatch chain**: per acp-ajudd#74 (┬з The three roles ┬з Notes, not inbox-writes), a coding session inside a live chain surfaces work by handing a **note up the chain** to planning rather than writing the inbox itself; the direct-write carve-out (acp-ajudd#10) survives only when there is no chain to route through (a solo session, or a cross-repo mailbox drop). Full statement: `references/inbox-convention.md` ┬з State-exclusivity.

**Completion authority тАФ only coding-finish closes implemented work (acp-ajudd#42).** "Complete" means *implemented*, so the completion stamp `[DONE]` is written **only by a coding session's `/session:finish`** (or a coding `checkpoint` closing its own picked-up work). A planning / refine / sessionless context may create, refine, delete, backlog, or set-aside work freely тАФ everything **except** mark it complete. It has two non-completion archive words instead of `[DONE]`: `[DISPOSITIONED тАж тАФ <fate>]` (read a capture, won't build it as-is) and backlog (defer). Distinct again from the coding-only **pickup** stamp `[CONSUMED <date> тЖТ session <name>]` (*taken, not done*). And a **parent/index entry closes bottom-up** тАФ never self-completed by planning while its implemented children are still open; each child is `[DONE]` by its own coding session first. Three stamps, three owners тАФ full table in `references/inbox-convention.md` ┬з Disposition & completion.

## Work ownership тАФ contract/requirements docs are developer-owned (extends acp-ajudd#13)

┬з State-Exclusivity makes a *live `work` entry* developer-owned тАФ a coding session consumes it, it never diverges from it. That same ownership extends to the other **contract/requirements artifacts** a coding session works next to but did not author: a **spec, ADR, design doc, or ticket body** тАФ all of which are just **work** in documented form. #13 already established that requirements are planning/developer-owned and that coding hands off rather than diverging; this section only names specs / ADRs / tickets тАФ which #13 hadn't тАФ as members of that same class. (It does not restate #13; read it there.)

**The rule.** Contract/requirements documents are **developer-owned work**. A coding session edits them **only on explicit developer direction** тАФ including the developer invoking a spec/refine flow тАФ and **never on its own initiative**, not even under a standing "keep the doc current" instruction. Live/ephemeral status тАФ dated events, test counts, blockers, the next action тАФ belongs in the **session file**, not the work doc: **link to the work, don't mirror it** (neither the work's content into the session nor the session's status into the work). This is phrased deliberately as a *class of document*, not a path тАФ there is no `specs/` or per-repo location baked in, so it holds in any project.

**Precedence tie-breaker тАФ who wins where they seem to overlap.** The work and the session own **separate territory**, so they cannot actually disagree on a reviewable fact:
- The **work** (spec / ADR / ticket) is authoritative for everything **reviewable** тАФ scope, acceptance criteria, done-criteria, decisions.
- The **session** is authoritative **only** for **live state** тАФ where the work is, blockers, next action тАФ and **never overrides the work doc.**

Duplicated status is the *only* thing that ever "drifts" (one fact copied into two files, then updated in just one тАФ the "5/5 vs 6/6" failure). Once status lives only in the session and reviewable facts only in the work doc, there is nothing left to conflict.

**Session entries are point-in-time snapshots (folds in the staleness concern).** Every session entry is **timestamped and explicitly not a live source of truth** тАФ it records what was true when written. Re-verify a session fact against the work doc and the code before relying on it. A stale entry can't mislead precisely because the session is never authoritative for a reviewable fact тАФ that is the precedence rule doing its job, not a session-format flaw.

**Authoring note тАФ committed session files are read by teammates (folds in the no-enforcement concern).** Session files are deliberately freeform prose (the record is the linted artifact тАФ the split is intentional, no lint on the journal). One convention: in any session file that commits (see ┬з The committed-sessions model), don't reference personal plugin commands (`/session:commit`, `/session:finish`, etc.) as if they were universal тАФ a teammate reading the committed file may not have the plugins installed. Describe the action plainly ("committed and pushed") or note it's plugin-driven. Convention only тАФ no enforcement mechanism.

**Instruction-only тАФ deliberately no hook.** This is a *judgment* problem (is a given edit illegitimate status-bleed, or a sanctioned developer-directed spec change?), not a pattern-matchable one. A hook could match only on file path тАФ a brittle per-repo list that would *also* block correct spec edits. The instruction rides in context whenever a session is active, so it reaches the AI at authoring time. Same rationale, and the same no-hook stance, as #13 and acp-ajudd#1.

## Captures Inbound (the `capture` type) тАФ human-driven

The inbox holds two types (acp-ajudd#62): it is both a **to-do list** (`work` at `new`/`refining`/`ready` you pick up and build) and an **inbound stream** (`capture`-type entries тАФ raw inbound one session drops for another: a heads-up, a payload of values, a stray idea). Reading captures is **human-driven** (acp-ajudd#10): Claude never polls, monitors, or auto-announces them. *(This was the "note/data mailbox"; the behavior is identical тАФ there are no note/data sub-labels, just the `capture` type awaiting disposition.)*

- **Send:** `/session:inbox` drops a `capture` (or a declared `work` handoff) into a target slug's inbox тАФ a free-rein write with a visible confirmation line in the sending session (per acp-ajudd#5). Nothing pings the recipient. The sender declares the type; when unspecified, Claude infers from content and confirms (┬з The three roles; `references/inbox-convention.md` ┬з Inbox Model). The old `intent:` sub-label (`story`/`fyi`/`data`) is retired тАФ the `work`/`capture` type carries the only distinction.
- **Surface:** a single **"Captures waiting: N"** line at `session:start` (and in the switch/resume blocks) when any `capture`-type entry exists. That is the only automatic surfacing тАФ one glance, not monitoring. Captures **never** appear in the pickup list or the checkpoint/finish sweeps.
- **Read тЖТ disposition тЖТ archive (on request only):** when the user says "check captures" / "read the capture from `<repo>`", read every `capture`-type entry in the slug inbox and disposition each тАФ **promote** (real work тЖТ becomes `work` at `refining`), or a read-and-archive fate: **discard**, **absorb into the current session** (fold an FYI or a data payload тАФ inline or via a `ref:` file тАФ into the work at hand), or **feed a refinement**. Then **archive** each non-promoted capture to `_inbox_archive.md` with the **bucket-3 planning-disposition stamp** `[DISPOSITIONED YYYY-MM-DD тАФ <fate>]` (`discarded`/`absorbed`/`refined`) and remove it from the live inbox (archived, never deleted; a promoted capture stays live as `work` at `refining`/`ready`). This is a **non-completion** stamp тАФ dispositioning a capture is neither completing implemented work (`[DONE]`, coding-finish only) nor a pickup-consume (`[CONSUMED тЖТ session]`). Full flow + the three-stamp table: `references/inbox-convention.md` ┬з Disposition & completion and ┬з Captures inbound.

## The three roles тАФ the dispatch model (acp-ajudd#57)

**Scope тАФ inbox zones only (plugin / personal). Work repos are explicitly OUT.** This whole section describes how **multiple live terminals on one machine** collaborate through a local `_inbox/` inbox and same-machine copy-paste. It applies **only where there is a local inbox to dispatch from** тАФ the plugin and personal zones. **Work repos stay exactly the plain two-move Jira flow:** `refine` in Jira (*Gathering Requirements* тЖТ *Ready For Work*), a dev `code`s the story. Jira **is** the queue and the system of record; there is no local inbox, and we do **not** build a dispatch layer over Jira cards. Everything below = inbox zones; work repos = the plain Jira refineтЖТcode flow.

┬з Session Stance splits a working context into two **stances** by file-presence: planning (sessionless) vs coding (has a session file). This section refines the *planning* stance into two distinct **roles** and names the coding stance's role, giving **three roles** total that collaborate in real time. Roles are the operational unit; stance is still the coarse 2-way split ({refine, dispatch} = planning-stance / sessionless; {code} = coding-stance / has a file).

| Role | Stance | Session file? | What it does | Token profile |
|------|--------|---------------|--------------|---------------|
| **refine** | planning | no | *scope work* тАФ edits inbox **`work`** only; the inbox author (mints IDs via `inbox-id.py`, writes/iterates `work`, promotes `capture`тЖТ`work`). Entered by the `refine` verb (acp-ajudd#56). | long-lived, lean, low-token тАФ retains reasoning across many entries |
| **dispatch** | planning | no | *coordinate work* тАФ pulls `ready` `work` entries, bundles related ones, sends implementation notes to `code`, validates the returned tree post-hoc, and **decides validated / safe-to-close** (completion itself is a coding-finish stamp тАФ ┬з Disposition & completion). A reader/coordinator: does **NOT** edit plugin code and authors **no** inbox entries **except clear, no-refinement splits of in-flight work** (acp-ajudd#95); it never refines requirements or mutates an existing entry. | long-lived, lean |
| **code** | coding | **yes** | *implement* тАФ folds in the `ready` `work` and builds. Fresh every implementation. Entered by the `code` verb. | ephemeral, token-heavy (a clean session per build is deliberate тАФ no giant planning history riding along) |

**Role determination тАФ told, not auto-detected (cheap to tell):**
- **`code` is self-evident from the file** тАФ a session file exists тЖТ coding (the file dictates the mode, acp-ajudd#56). No telling needed.
- **`refine` is the default** sessionless stance тАФ start with no session file and you are refining (planning has always been the default тАФ ┬з Session Stance).
- **`dispatch` is explicitly assumed** тАФ it cannot be told apart from `refine` by file state (both sessionless), so it is entered by *being told*. The first-class way to tell it is **`/session:dispatch`** (`commands/dispatch.md`, acp-ajudd#71): invoking the command **is** the "told" тАФ it declares the dispatch role, loads the discipline, and orients on the inbox, with no briefing-paste needed. Dispatch is **surfaced on the start screen** too, as a **secondary/advanced line beneath the primary two verbs** (refine / code) in the **plugin/personal** zones only тАФ the two-verb headline is unchanged (acp-ajudd#56); dispatch is a quiet third option in-zone, absent in work/general where the model doesn't apply. **Briefing-paste is demoted to a documented cross-machine fallback** тАФ still valid when a dispatch posture must be adopted in a terminal that can't run the command, but no longer the primary bootstrap.
- Each session therefore **knows and declares its own role**: file тЖТ `code`; told (via `/session:dispatch`, or a pasted briefing on another machine) тЖТ `dispatch`; else тЖТ `refine`. When it emits a handoff it stamps that role as the `from-role` (┬з Cross-Session Paste Handoff).

**Two channels тАФ the decoupling that makes it non-blocking:**
- **refine тЖТ dispatch is BOTH the inbox AND a planning handoff (acp-ajudd#71 тАФ corrects #57's inbox-only framing).** How dispatch gets its marching orders depends on whether planning and implementation happen **concurrently** or **separated in time** тАФ two **situational, co-equal modes**, not a primary + fallback:
  - **Deferred тАФ dispatch-autonomous from the inbox** (async, pull-based): `refine` drops `ready` `work`; a later `dispatch` context pulls when free, working out order and grouping itself from the inbox and its `depends-on` markers (acp-ajudd#67). No live planner to direct it тАФ **the inbox IS the refineтЖТdispatch interface.** This is what lets refine keep scoping the next thing while dispatch drives the current one.
  - **Concurrent тАФ planning-directed** (push-based): when planning and implementing run in the same stretch, the planning session holds the live context, so it **computes order + grouping and hands dispatch an explicit plan** via a `/session:handoff` (it just scoped the work and knows the dependencies). Dispatch acts on that pasted plan rather than re-deriving it. *(Example 2026-07-13: planning sequenced #66 тЖТ #67 тЖТ #65, marked the `depends-on`, and handed it over.)*
  Both are first-class; the user picks by how they're working. `commands/dispatch.md` ┬з Choose the Operating Mode owns the runtime detail.
- **Copy-paste notes тЖТ dispatch тЖФ code** (real-time, push-based). Faster than the inbox, and lets a low-context coding session be handed exact steps / watch-fors / report-back rules.

**Strict hub topology тАФ dispatch is the sole relay + validator; planning and coding never exchange notes directly (acp-ajudd#74, tightens #57).** #57 said "any role тЖТ any role is legal"; live experience tightened that into a **strict hub cycle.** The only legal edges are **planning тФАтФАтЦ╢ dispatch тФАтФАтЦ╢ coding** and **coding тФАтФАтЦ╢ dispatch тФАтФАтЦ╢ planning**. There is **no direct planningтЖФcoding edge in either direction** тАФ *within the hub.* (A direct `planning тФАтФАтЦ╢ coding` handoff is still **possible**, but it is deliberately **not** a hub edge: it starts a **SOLO** coding session whose report-back is a courtesy note outside the hub тАФ see the "Dispatch-always is the NORM" paragraph below, Pillar 4, and ┬з The dispatchтЖФcode loop's solo carve-out. So this absolute governs *hub* edges, and the solo path ~140 lines on is a carve-out, not a contradiction.) Dispatch sitting in the middle тАФ relaying notes *both* up to planning and down to coding тАФ is exactly what makes its reason-for-being real: it **absorbs the build loop so planning stays in flow.**
- **Two-hop relay.** A codingтЖТplanning thought is never pasted into planning. Coding emits a `CODING тФАтФАтЦ╢ DISPATCH` note; dispatch re-emits a `DISPATCH тФАтФАтЦ╢ PLANNING` note. Two hops, each titled with its **own** unambiguous destination.
- **The two-ended title is the routing instruction, not decoration (acp-ajudd#69).** Because the human courier must know which of the two terminals each note goes to, the `<FROM-ROLE> тФАтФАтЦ╢ <TO-ROLE> HANDOFF` title (┬з Cross-Session Paste Handoff) carries the destination the courier acts on: a note is "placed" wherever its `тФАтФАтЦ╢ <TO-ROLE>` points. The title (sender-side) pairs with the receiving-side target check (┬з Receiving side тАФ verify the target before acting) as the two guards on the one human-relay mispaste risk.
- **Happy path is silence** тАФ if refinement is good enough, dispatch just forwards and code just implements, nothing said.

**Dispatch-always is the NORM тАФ planning routes work through dispatch, not straight to coding (acp-ajudd#75, affirms #74).** The default and expected path for getting `ready` work built is **planning тФАтФАтЦ╢ dispatch тФАтФАтЦ╢ coding**: even a trivial item routes through dispatch, because dispatch is what supplies the model's core value тАФ an **independent validator who did not write the code** (┬з The dispatchтЖФcode loop, leg 4). A **direct planning тФАтФАтЦ╢ coding handoff is NOT a sanctioned hub edge** тАФ it is the deliberately-narrow **solo** case, reframed (not blessed as a new edge) in Pillar 4 and ┬з The dispatchтЖФcode loop's solo carve-out: a planning-initiated direct handoff starts a **solo** coding session whose report-back is a **courtesy** note *outside* the strict hub, never a topology-legal relay. The load-bearing cost of any such bypass: **no independent validator runs** (the coding session verifies only its own work), so a bypass is safe **only for doc-only work with crisp Done-whens** тАФ anything behavior-changing wants the dispatch hub and its second pair of eyes. (A third option тАФ adding a real exception edge to the hub for direct handoffs тАФ was considered and **rejected**: the hub stays strict; the bypass is a solo session, not an edge.)

**One inbox author тАФ the real meaning of the acp-ajudd#30 cap.** #30's "one planning session per repo" was really guarding the **shared ID counter and the single `_inbox.md`** against two concurrent *requirements* writers. Its intent survives as **one *requirements* author among planning contexts тАФ `refine`** (the sole concurrent-planning writer of requirements; a **solo / no-chain coding session may still post entries** too тАФ rule 4 and the acp-ajudd#10 carve-out тАФ so this is not "only refine ever writes," it is "only one *requirements* writer"). `dispatch` is a *second sessionless context, and that is legal* precisely because it is a **reader/coordinator that authors no requirements** тАФ the one thing it *may* write is a clear, no-refinement **split** of in-flight work (acp-ajudd#95), and even that is minted through the atomic `inbox-id.py` and never touches an existing entry, so it cannot race the counter or fork a live item. (**acp-ajudd#102 hardened the file side structurally:** the consolidated inbox is now one file per item тАФ `_inbox/<id>.md` тАФ so two writers on *different* items touch *different* files and cannot clobber each other at all; the "one requirements author" discipline is now about coherence of the requirements set, not about avoiding a shared-file write race.) (The counter race is also closed in code by acp-ajudd#31's atomic minting in `inbox-id.py`.) So the cap is not "one sessionless session"; it is "one *requirements* author of entries" тАФ refine тАФ plus coordinators that write only no-refinement splits. The coding cap (one `code` session тАФ one writer of `_active` / `_index` / the working tree) is unchanged.

**Token rationale (the "why").** refine and dispatch stay **long-lived and lean** (they carry reasoning, not implementation churn); `code` stays **fresh and ephemeral** (a clean session per build, no planning history dragging along). Decoupling *implement-progress* from *scope-progress* is the whole point: being mid-refinement never stalls the dispatchтЖФcode loop, and a heavy build never bloats the planning context.

**Inbox write-authority & work-entry immutability (acp-ajudd#59, refined by #95).** The three roles have distinct, non-overlapping authority to *write* the inbox тАФ a follow-up to #57 that spells out the write rules the model left implicit (instruction/doc-only, no hook тАФ acp-ajudd#1). acp-ajudd#95 later **relaxed the dispatch line from *never writes* to *never writes requirements*** тАФ dispatch may author clear, no-refinement **splits** of in-flight work (rule 3's named exception), the companion to the #94 loop-cleanup (ship тЙа close) so the write-authority and the close model read consistently. Four rules, one set:

1. **Consumed = frozen (hard line).** Once `code #X` converts a `work` entry (fold-then-archive тАФ ┬з State-Exclusivity), it is **frozen: no writes to it after conversion, by any role.** It is gone from the live inbox, and the archived `[CONSUMED тАж тЖТ session <name>]` copy is history, not an editable record. A change the build turns out to need becomes a **new `work` entry / story** тАФ never a reopen of the consumed one. Rationale: implementation moves fast and nothing waits on the retired entry. (Hardens acp-ajudd#13/#40.)
2. **`refine` = the requirements author, writing surgically.** The owner of *requirements* authoring in the inbox (┬з One inbox writer): it creates / edits / archives entries with **targeted, single-entry edits**, sets each entry's `work`/`capture` type, and is the **only** role that defines or refines what an entry asks for. (Dispatch may author *clear, no-refinement splits* of in-flight work тАФ rule 3's exception тАФ but never refines requirements; that stays refine's alone.) It **never** does a bulk cross-entry rewrite тАФ with per-item storage (acp-ajudd#102) each edit touches exactly one `_inbox/<id>.md` file, so the whole-file rewrite that misfiled an entry during the #57 pilot (a refinement-side script, not a coding or dispatch violation) is now structurally impossible, not merely discouraged.
3. **`dispatch` = read-only + notes-only, with ONE named exception тАФ no-refinement splits (acp-ajudd#95, revises #59).** It reads and bundles `ready` `work` and feeds handoff notes to `code`; as a rule it **never** analyzes requirements or audits code for correctness (that is `code`'s job тАФ dispatch only validates the returned tree against the entry's Done-whens), and it **never mutates, forks, or writes back to an existing entry** (consumed = frozen + item-immutability hold тАФ rule 1 and ┬з State-Exclusivity). **The exception:** dispatch **MAY author a NEW `work` entry directly when it is a split of, or a follow-on within, the work it is currently coordinating AND it needs no refinement** (clear, ready-to-run) тАФ minted via `inbox-id.py`, stamped `from <slug> / dispatch`. This is **coordination, not requirements-definition**: splitting in-flight work into ready sub-tasks is dispatch's job; **defining / capturing / refining requirements is NOT** (that is planning's, via `refine`). Two hard limits ride the exception тАФ dispatch authors only **splits / follow-ons within the scope of the work in hand** (never an unrelated new item; out-of-scope ideas go to planning or a capture), and **if a split needs refinement it hands it to `refine`** (a note routing the authoring up; planning refines it and hands it back `ready` тАФ the two-jump path, reserved for the case that actually needs it). Being a reader/coordinator that writes only clear splits тАФ never racing the requirements author on an existing entry, every write minted through the atomic `inbox-id.py` тАФ is why a second sessionless context stays legal (┬з One inbox writer). *(This is why dispatch authored the #86тАУ#93 Confluence-refresh splits directly in the live 2026-07-14 run тАФ a split of in-flight coordination, needing no refinement.)*
4. **`code` = unrestricted (pre-conversion).** It may edit a *live* `work` entry freely before conversion (the plan-then-code-in-one-session path тАФ ┬з State-Exclusivity); it consumes/archives the entry at conversion, after which rule 1 freezes it. No lockdown.

**Notes, not inbox-writes тАФ the inbox is planning's single write-surface (acp-ajudd#74, Pillar 2).** Inter-session communication is handoff **notes**; the inbox is written by **`refine`/planning** only. A **coding session that is part of an active chain NEVER writes inbox entries** тАФ when it has something to surface (a follow-up, a forward suggestion, a found issue) it hands a **note up the chain** (`coding тФАтФАтЦ╢ dispatch тФАтФАтЦ╢ planning`) and planning decides what becomes an entry. This kills the cross-context capture race (the near-duplicate #72/#73 incident) **structurally, not by discipline** тАФ funnelling requirements/capture authoring through planning means a chained coding session cannot mint racing entries. It sharpens rule 4 and the "posts NEW inbox entries" clause of ┬з State-Exclusivity: those free writes belong to a coding session that is **not** part of a live dispatch chain. *(acp-ajudd#95's dispatch-split exception does not reopen this race: dispatch's splits are planning-side, always atomically minted via `inbox-id.py`, and only ever NEW entries тАФ never a fork of a live one тАФ so "planning is the single requirements write-surface" still holds.)*
- **Carve-out тАФ keep acp-ajudd#10 (captures-direct only when there is no chain to route through).** A direct capture-to-inbox stays legal **only when there is NO active chain**: a cross-repo mailbox drop (a work-repo session parking a thought in the plugin inbox), or a **solo** coding session with no dispatcher. No note channel exists in those cases, so the direct drop is the only option. **Rule in one line: notes when a chain is live; direct capture only when there is no one to hand a note to.**

**Not a concurrency lock.** With dispatch read-only, `refine` and `code` remain the only inbox writers, and the sequential handoff flow (one session active on a given entry at a time) plus surgical single-entry edits are what keep writes safe тАФ there is no mutex, no hook, no policing (consistent with acp-ajudd#1). Like the rest of the dispatch model, this is **inbox-zone only** (plugin / personal); work repos stay the plain Jira flow with no dispatch layer.

**Dispatch operating discipline тАФ communicate only via notes; validate, don't punt (acp-ajudd#63).** The three-role pilot surfaced dispatch behaviors #57/#59 left implicit: during a live run, the dispatch context kept bouncing decisions back to the human ("holding for your review", "ship or fix?") when it should simply have produced a handoff note. Four rules pin the behavior down (instruction/doc-only, no hook тАФ acp-ajudd#1):

1. **Handoff notes are dispatch's ONLY inter-session channel.** Dispatch talks to the `code` session and the `refine` session through paste-block handoff notes тАФ nothing else. It does **not** ask the human questions or seek the human's approval; the human only *relays* paste-blocks between terminals and is **not a decision endpoint** for dispatch. (Dispatch may still narrate to the human what it is doing тАФ the ban is on routing *decisions / questions / approvals* to the human, not on talking.)
2. **Dispatch validation REPLACES a human review-gate.** Dispatch validates returned work itself тАФ the actual working tree / files against the entry's Done-whens (┬з The dispatchтЖФcode loop) тАФ and *that* validation IS the check. It does **not** escalate a "review" to the human. For an outward-publish / hard-to-reverse item, dispatch's clean validation is the greenlight, emitted as a "publish & complete" note to `code` тАФ it does not punt the review to the human.
3. **Blocked тЖТ a note to PLANNING (`refine`), never a question to the human.** If dispatch hits something it cannot resolve by reading тАФ a genuine blocker, an ambiguity, a cross-item conflict тАФ the resolution is a handoff note to the `refine` role (the spec owner), which answers or re-scopes. Questions get answered by `code` or `refine`, never by the human.
4. **Dispatch stores nothing in files** (read-only + notes-only, per acp-ajudd#59 тАФ ┬з Inbox write-authority rule 3; the lone exception is authoring a clear, no-refinement **split** of in-flight work тАФ acp-ajudd#95) but **reads anything and everything freely.**

**Validation is Done-when-checking, not design re-derivation тАФ the intelligence lives in the Done-whens (acp-ajudd#74, Pillar 3).** Dispatch validates by checking the **produced tree against the entry's acceptance criteria** (the **Done-whens**, authored by `refine`) plus the diff тАФ it does **NOT** re-derive the design and needs **none** of planning's design context. It is *smart enough to check a written contract, dumb enough to escalate anything the contract does not cover.* **The burden is on `refine` to write good Done-whens**, not on the dispatch session to be clever тАФ which is why a low-context dispatch context can validate correctly (proven live 2026-07-13: #66 validated with zero design context тАФ a peek check + convention-doc check + bump check). **Routine validation therefore NEVER reaches planning.** Residual (accepted, per #57's non-gating): incomplete Done-whens can let a miss through; it surfaces later as a follow-up costing one extra deploy тАФ not gated. The **rare** codingтЖТplanning case is an **escape-hatch escalation flagged by coding and relayed by dispatch** (never a direct edge): coding produces a note flagged for planning (`State: FOUND-ISSUE` / `REQUIREMENTS-CHANGE`), dispatch **relays it up** without judging its relevance тАФ coding declares it. Planning is interrupted **only** for a genuine design decision.

**The one human gate is keyed on REVERSIBILITY (refines rule 2).** Rule 2 above ("dispatch validation replaces a human review-gate") holds for internal deploys **and** for editable/recoverable outward channels тАФ but **not** for irreversible ones:
- **Teams messages are IRREVERSIBLE** (can't unsend or edit) тЖТ **human PRE-APPROVAL is required** before posting (draft-before-send: the human reviews the exact chat and the exact content тАФ ┬з Teams Messaging). This is the **one hard gate** dispatch/coding routes to the human.
- **Confluence / Jira / GitHub are editable/recoverable** тЖТ **publish without pre-approval, but MUST auto-open the page / PR** for post-hoc review (ties to the standing "auto-open links/PRs" preference); fix in place if the post-hoc look finds something.

So dispatch greenlights everything **except a Teams send**, which always goes to the human first. This **corrects the earlier "Aaron reviews before publish" framing** (e.g. the acp-ajudd#60 Confluence-refresh spec, which mis-stated a draft тЖТ human тЖТ publish gate): Confluence **publishes + auto-opens**, no pre-gate тАФ the reviewer is dispatch, greenlighting via a note; only an irreversible Teams send pre-gates to the human. Extends acp-ajudd#59 (dispatch read-only + notes-only) with this behavioral layer.

**Dispatch checks prerequisites before pulling тАФ an unmet `depends-on` means NOT dispatchable (acp-ajudd#67).** Some `work` must land in order, and that ordering is now **machine-visible** on the entry itself: an optional `> [depends-on: <id> тАФ <reason>]` line under the header (`references/inbox-convention.md` ┬з Sequencing). Before dispatch pulls or bundles a `ready` entry it reads that line: a cited prerequisite is **met** only if that entry is `[DONE]` or `[CONSUMED тЖТ session тАж]` (implemented, or in-flight in a coding session), and **unmet** if it is still live (`new`/`refining`/`ready`) or itself blocked. An entry with any unmet dependency is **held, not dispatched** тАФ dispatch skips to the next dispatchable entry and, if the ordering is unclear or contested, routes a note to `refine` (never a question to the human, per rule 3 above). This closes the gap the acp-ajudd#60 detour exposed (#60 was dispatched before its prerequisite #62 had shipped, then had to be stood down тАФ ┬з HALT). It is the reader half of the `depends-on` marker `refine` writes, and it is what makes dispatch's **autonomous-from-inbox** mode reliable (┬з The three roles; `commands/dispatch.md`). Instruction-level only тАФ no hook (acp-ajudd#1).

**Orchestration is DETECTED, not declared тАФ the keystone (acp-ajudd#74, Pillar 4; sharpened by #75).** A coding session knows it is a coding session because it **has a session file** (the file dictates the mode); it has **no** knowledge of a dispatch/planning layer above it. It learns it is **orchestrated ONLY by receiving a handoff note FROM DISPATCH** (acp-ajudd#75 sharpens #74's "a handoff note" to "a note *from dispatch*" тАФ the **sender's role** is what puts the session in the hub, since dispatch is the sole relay + independent validator). That single fact settles the whole return-handoff question, in three cases:
- **No handoff note ever received тЖТ solo** тЖТ self-verify, ship, **no report** (there is no one to report to тАФ the existing solo carve-out, unchanged).
- **A note FROM DISPATCH тЖТ orchestrated** тЖТ **ALWAYS emit a return handoff** to dispatch (`State: IMPLEMENTED-DEPLOYED`) тАФ but **at the ship (build-end), not at close** (acp-ajudd#94). The happy path is *build тЖТ deploy тЖТ report `IMPLEMENTED-DEPLOYED`*; the session then stays **active** and does **not** run `/session:finish`. The return block is a topology-legal `coding тФАтФАтЦ╢ dispatch` relay and is what makes acp-ajudd#72 facet-1 precise: "coding-finish always hands a note back" means *always, IF fed by a dispatch note* тАФ now sharpened to *the ship hands the note back; the close comes later.* The detection cue is that the session was created from a **dispatch** work order (recorded in its "Picked up from inbox" provenance as a `dispatch тФАтФАтЦ╢ coding` handoff). The **close** (`/session:finish`, the only thing that flips `completed`) fires afterward on the validated signal (`SAFE-TO-CLOSE`, or an `Action: CLOSE` note) тАФ ┬з The dispatchтЖФcode loop, leg 4тА▓.
- **A note DIRECT FROM PLANNING (dispatch bypassed) тЖТ solo, NOT orchestrated (acp-ajudd#75).** A `planning тФАтФАтЦ╢ coding` direct handoff does **not** put the session in the hub тАФ there is no dispatch relay, so no `coding тФАтФАтЦ╢ dispatch` return exists. The session runs **solo** (self-verify, finalize on its own authority, like a bare `code #X`); its report-back to planning is a **courtesy** note explicitly **outside** the strict hub тАФ a solo report, *not* a topology-legal relay, so it is **not** the direct planningтЖФcoding edge #74 forbids (that ban is on *hub* edges; a solo courtesy report is a different animal тАФ ┬з The dispatchтЖФcode loop, solo carve-out). Cost, as everywhere: no independent validator runs, so this path is for **doc-only work with crisp Done-whens** only.

`commands/finish.md` owns the mechanics.

**The happy-path loop TERMINATES AT DISPATCH тАФ no completion report up to planning (acp-ajudd#84, revises #74 / #72 facet 4).** On a successful, validated completion, dispatch does **NOT** send a `dispatch тФАтФАтЦ╢ planning` completion handoff. It does two things and stops: **(a)** shows the **human** the `SAFE-TO-CLOSE` close-signal (voiced as a courier line тАФ ┬з The courier line) so the coding terminal can be cleared; and **(b)** **pulls the next item itself.** Sequencing needs no report-up тАФ dispatch already holds the plan (planning-directed mode) or reads the inbox with its `depends-on` markers (autonomous mode), so it releases the next work order on its own. This corrects #74's earlier "completion handoff back to planning at each batch," which pulled planning into a validation/report loop it did not need тАФ the exact interruption dispatch exists to prevent (a live `VALIDATED` report cost a planning turn with nothing to do, 2026-07-13). The happy path is therefore a two-role loop: `coding тФАтФАтЦ╢ dispatch` (finish return) and `dispatch тФАтФАтЦ╢ coding` (next work order), plus the human close-signal тАФ planning stays out of it.

- **`dispatch тФАтФАтЦ╢ planning` is ESCALATION-ONLY.** The sole traffic up to planning is a genuine escalation that needs planning's design authority тАФ a coding-flagged question / unclear / disagreement / found-problem / requirements-change that dispatch relays up (the escape-hatch, ┬з Validation is Done-when-checking; coding declares it via `State: FOUND-ISSUE` / `REQUIREMENTS-CHANGE` and dispatch relays without judging its relevance). Routine validated completions never reach planning.
- **What STAYS (do not overcorrect тАФ decision A):** dispatch still **fully validates the returned tree against the entry's Done-whens** тАФ the independent-validator payoff (┬з The dispatchтЖФcode loop, leg 4) is unchanged; only the *report-up on success* is removed, not the validation. And the **dispatch тЖТ human roadmap status glance stays** тАФ where the whole roadmap stands (waves done / in-flight / queued, plus `refining` and captured items) is for the *person* at a glance, not a note to planning; emit it at batch kickoff. `commands/dispatch.md` owns the cadence.

### The ideation station тАФ `capture`, upstream of the three roles (acp-ajudd#96)

The three roles (refine / dispatch / code) turn a *scoped* idea into shipped work. **`capture` sits one step further upstream** тАФ it is the **ideation station that feeds `refine`**, not a fourth member of the trio (the section name and its many cross-references stay stable тАФ this is deliberately **additive, not a rename**). The full pipeline:

```
  capture  тФАтЦ╢  refine   тФАтЦ╢  dispatch   тФАтЦ╢  code
 (ideation)   (planning)  (coordinate)  (implement)
 sessionless  sessionless  sessionless   session-backed
```

**Why it exists тАФ the last bottleneck.** The multi-window model removes one bottleneck per split: `code` split from planning (plan the next thing while code runs), then `dispatch` split out (stop interrupting planning to feed the coder тАФ acp-ajudd#57). The residual: **banking a new idea forces you to interrupt whatever window is busy.** The only in-zone way to record one was `refine` тАФ full planning (explore, scope, produce `refining`/`ready` work), far too heavy for "jot this for later." `capture` is a free-sitting window you fire ideas at anytime without touching the other three; the thesis is **invert the wait** тАФ the windows wait on the human, not the reverse.

**Sessionless, plugin/personal only, writes `capture`-type entries.** Like `refine` and `dispatch`, capture is a planning-stance context with **no session file** and no `_active` touch. It applies **only in the inbox zones** (plugin / personal) тАФ a work/general repo has no local inbox, so a stray idea there leaves via a `/session:inbox` capture aimed at a repo that has a system of record. Its **only durable output** is a `> [type: capture]` entry (the *capture type* тАФ ┬з Captures Inbound), minted via the atomic `inbox-id.py` and written as its own `_inbox/<id>.md` file (acp-ajudd#102); it emits **no handoff blocks** (it is outside the planningтЖФdispatchтЖФcoding relay).

**The load-bearing boundary тАФ triage vs. spec, NOT read vs. no-read.** Capture and refine are separated by **output and commitment, not tool access** тАФ both may read the repo:
- **Capture answers "is this worth planning's time?"** тАФ a viability gate; it **MAY read/search code** to answer that, so planning's hopper isn't flooded with non-viable ideas.
- **Refine answers "here's exactly what to build."** тАФ a scoped spec.

So capture **MAY** read/search code, assess rough difficulty/cost-benefit, and give a **one-line** viability verdict; it **MUST NOT** edit or write code, produce a design or scoped spec, set any status past `capture`, or "start refining" тАФ the hard rule mirrors "refine never offers to code" as **"capture never refines."** A **depth guard** keeps the peek a *sniff*: if "is this viable?" turns into a deep read, capture notes "worth a proper look" and **stops** тАФ the hard cap is on **output** (a note or nothing, never a spec). Two exits only: **write a `capture` entry** (raw idea + one-line viability note) or **drop it** (a cheap death is a win). The viability note earns the read-access тАФ it is an **artifact** that lets `refine` pick the capture up **pre-warmed** instead of cold.

**How the loop closes тАФ bare `refine` drains the hopper.** Capture fills the inbox with `capture`-type entries; the **`refine` window's bare-`refine` path surfaces "Captures waiting: N тАФ say 'check captures' to triage them"** so planning naturally drains what capture filled. Reading/dispositioning captures is the human-driven flow already defined in ┬з Captures Inbound and `references/inbox-convention.md` ┬з Captures inbound (acp-ajudd#10) тАФ promote to `work` (`refining`), or discard/absorb/feed. Capture never dispositions its own entries; **writing and draining are different stations.**

**Concurrency-safe as a 4th window.** Capture only **appends new items** тАФ it never forks or mutates an in-flight entry (consumed = frozen, item-immutability hold тАФ ┬з State-Exclusivity). It is safe as a fourth concurrent window because the atomic ID mint already landed (acp-ajudd#31 atomic mint, #66 self-heal); a pre-#31 fourth window appending IDs would have raced. `commands/capture.md` owns the mechanics.

## Cross-Session Paste Handoff (the handoff block)

Three paths move work between sessions, and they split on **how the item travels**:

| Path | Mechanism | Crosses machines? | Command |
|------|-----------|-------------------|---------|
| **inbox** | writes a capture into a target slug's `_inbox/` (one file per item) | no тАФ same machine, file-based | `/session:inbox` |
| **spawn** | writes a `[spawn]` inbox item staging a linked follow-on session | no тАФ same machine, file-based | `/session:spawn` |
| **handoff** | prints a self-contained block the human **copies and pastes** into another live terminal (another Claude Code conversation / another machine) | **yes** тАФ human-carried | `/session:handoff` |

`inbox`/`spawn` hand off *through the filesystem* тАФ the next `/session:start` on this machine surfaces them. `handoff` is the **human-carried** case: the block is copied out of this terminal and pasted into a different live terminal that has none of this conversation's context. That is why the block must be **self-contained**.

**`handoff` has two forms** (acp-ajudd#29), chosen by whether a coding session is active. From a **coding session** it produces the paste block **text only** тАФ nothing is written or sent. From a **sessionless planning context** it *also* writes **one durable local resume file** (`_context_planning-<topic>.md`, picked up by a fresh `/session:restore`) as the **primary** artifact тАФ paste is friction and lost with the clipboard, and planning тЖТ coding is the crossing that most needs to survive a restart. Even then it sends nothing and touches no `_active` / session file / `_index` тАФ writing those would be an in-place planningтЖТcoding conversion, which ┬з Session Stance (acp-ajudd#32) forbids. `commands/handoff.md` owns the two-path detail; this section owns only the block format below.

**The standard handoff block format** тАФ every produced handoff (by `/session:handoff`, or any time a session emits one) uses exactly this shape, and it is **role-aware** (┬з The three roles тАФ the dispatch model): the header names the sender's and receiver's roles, and adds machine-legible `Action` / `State` fields plus a human-facing close-signal. The header mirrors an **inbox-item header's provenance** (`[date @handle] from <source> (<zone>)`) so the receiver sees *who is handing to whom* without asking; horizontal rules give the block vertical breathing room and the body reads like a Teams message, not a wall:

````
тХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХР <FROM-ROLE> тФАтФАтЦ╢ <TO-ROLE> HANDOFF тХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХР
 [YYYY-MM-DD HH:MM @handle]   <from-role> (<from-session-name>) тФАтФАтЦ╢ <to-role> (<target>)
 Re:      <topic>  ┬╖  <inbox IDs if any, else omit the ID clause>
 Action:  <outbound intent тАФ PICK UP #57 / PICK UP #56,#57 (bundle) / FIX / VALIDATE / CLOSE / HALT>   тЖР on outbound notes
 State:   <return leg тАФ IMPLEMENTED-DEPLOYED / VALIDATED / FOUND-ISSUE / REQUIREMENTS-CHANGE / BLOCKED-QUESTION / HALTED>   тЖР on return notes (use instead of Action)
 Slug:    <slug>  ┬╖  Zone: <plugin|personal|story|cab|general>
тФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФА

 <self-contained body тАФ human-readable paragraphs, like a Teams message, not a wall>

тФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФА
 END HANDOFF ┬╖ [YYYY-MM-DD HH:MM @handle] ┬╖ <from-role> (<from-session-name>) тФАтФАтЦ╢ <to-role> ┬╖ <ids, else topic> ┬╖ <Action/State value> ┬╖ <return instruction тАФ see footer rules> ┬╖ <close-signal тАФ dispatch validation/close legs only, else omit>
тХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХРтХР
````
(The `HH:MM` in both stamps comes from a real `date` call at emit time тАФ see the datestamp field rule below. The footer is ONE compressed line тАФ the state recap, acp-ajudd#82.)

**Role-aware provenance header fields** (the meaningful part тАФ mirror the inbox-entry header). The handoff **role token** (used in both the title and the `тФАтФАтЦ╢` provenance line) is one of **{PLANNING, DISPATCH, CODING}** тАФ uppercased in the title, lowercased in the provenance line. **`PLANNING` is the canonical planning-side token and denotes the `refine` / `refinement` role** (the spec-owner planning role); `REFINEMENT`/`refinement` is an accepted synonym in older blocks, but new handoffs use `PLANNING`, and it is `PLANNING` that the acp-ajudd#69 receiver-side check matches a `refine`/`refinement` context against. (The three *roles* are `refine` / `dispatch` / `code` тАФ ┬з The three roles; the planning-side routing token collapses refine's two names to the single `PLANNING` so a `<to-role>` match is never ambiguous. `dispatch` is its own token тАФ `DISPATCH` тАФ even though it is also planning-stance; `code`/`coding` тЖТ `CODING`.) The fields:
- **Title (`<FROM-ROLE> тФАтФАтЦ╢ <TO-ROLE> HANDOFF`) тАФ names BOTH ends (acp-ajudd#69).** The block title carries the full direction so the human courier sees it at a glance, before reading a word of the body: `DISPATCH тФАтФАтЦ╢ CODING HANDOFF`, `CODING тФАтФАтЦ╢ DISPATCH HANDOFF`, `DISPATCH тФАтФАтЦ╢ PLANNING HANDOFF`, etc. `<FROM-ROLE>` is the origin role uppercased and **always matches the left side of the `тФАтФАтЦ╢` provenance line** directly below it (acp-ajudd#45); `<TO-ROLE>` names the destination. **The `<TO-ROLE>` is the routing instruction, not decoration** тАФ a note is "placed" wherever its `тФАтФАтЦ╢ <TO-ROLE>` points, and in the strict-hub topology (┬з The three roles, acp-ajudd#74) where dispatch relays notes both up and down, that destination is exactly what tells the courier which of the two terminals to paste into. It is the **sender-side** half of the mispaste guard, pairing with the **receiver-side** target check below тАФ and it is voiced to the human by the `тЦ╢ PASTE TO <ROLE>` courier line above the block (┬з The courier line). (Legacy single-ended titles тАФ `DISPATCH HANDOFF`, `PLANNING HANDOFF`, or the generic `SESSION HANDOFF` тАФ are historical and still read fine; the `<TO-ROLE>` addition is back-compatible and independently parsed. No migration.)
- **`[YYYY-MM-DD HH:MM @handle]`** тАФ the stamp an inbox item carries, **plus the wall-clock time (acp-ajudd#82)**. With three live windows (planning / dispatch / coding), the `HH:MM` is what lets the human reconstruct the sequence of handoffs across terminals. **The time is obtained by running a real `date` call at emit time** (e.g. `date +"%Y-%m-%d %H:%M"`) тАФ the emitting session does not otherwise know the wall clock, so a stamp composed without the call is a guess/hallucination and is never acceptable. The block spec *runs `date` at emit*; the same stamp is then echoed verbatim in the footer recap (below).
- **`<from-role> (<from-session-name>) тФАтФАтЦ╢ <to-role> (<target>)`** тАФ who тЖТ who, with each side's **role token** (planning / dispatch / coding тАФ lowercased here, uppercased in the title). This is the "where it came from" provenance. A sessionless origin is `planning (<slug>)` (the `refine`/`refinement` role тАФ canonical token `planning`; `refinement (<slug>)` is an accepted synonym) or `dispatch (<slug>)` (whichever role that context declared); a coding origin is `coding (<session-name>)`.
- **`Re:`** тАФ the topic, plus any inbox IDs the handoff carries (omit the ID clause when there are none).
- **`Action:` (outbound intent)** тАФ what the sender wants the receiver to *do*: `PICK UP #57`, `PICK UP #56,#57 (bundle)`, `FIX`, `VALIDATE`, `CLOSE`, `HALT`. Present on outbound notes (dispatchтЖТcode work orders, dispatchтЖТrefine change requests). `HALT` is the **stand-down** action тАФ stop the dispatched work cleanly mid-flight (┬з HALT). **`CLOSE` is the close-order (acp-ajudd#94)** тАФ dispatch tells a shipped-but-still-open coding session "validated тАФ run `/session:finish`." It is the note-form of the human-facing `SAFE-TO-CLOSE` signal, and it reliably triggers the full all-or-nothing close (┬з The dispatchтЖФcode loop, leg 4тА▓; `commands/finish.md`). Omit on pure return legs.
- **`State:` (return leg)** тАФ what *happened*, reported back: `IMPLEMENTED-DEPLOYED`, `VALIDATED`, `FOUND-ISSUE`, `REQUIREMENTS-CHANGE`, `BLOCKED-QUESTION`, `HALTED`. Present on return notes; used **instead of** `Action`. A note carries one or the other тАФ `Action` going out, `State` coming back. `HALTED` is the **stood-down** return state тАФ the work was stopped mid-flight before it finished, nothing published or committed (┬з HALT).
- **`Slug` / `Zone`** тАФ so the receiver knows the repo and project type without asking.
- **Close-signal (`SAFE-TO-CLOSE` / `HOLD`) тАФ human-facing, emitted by dispatch.** A terse token dispatch shows **the person** so a session can be killed at a glance тАФ it needs no processing on any other end. `SAFE-TO-CLOSE` means "no issues, move on"; `HOLD` names what to look at. It fills the **close-signal slot of the footer recap** on a dispatch validation/close note (other roles' notes omit that slot), and it is voiced to the human as a **courier line** from the fixed vocabulary (┬з The courier line below тАФ `SAFE-TO-CLOSE` maps to the `тЬЕ` line, `HOLD` to the `тП╕` line).
- **Footer тАФ a self-contained one-line "where I left off" state recap (acp-ajudd#82).** The footer is the **last thing visible** in each terminal window, so it must fully orient the human **without scrolling up**: one compressed, scannable `┬╖`-separated line echoing the key header state тАФ `END HANDOFF ┬╖ [YYYY-MM-DD HH:MM @handle] ┬╖ <from-role> (<from-session-name>) тФАтФАтЦ╢ <to-role> ┬╖ <ids, else topic> ┬╖ <Action/State value> ┬╖ <return instruction> ┬╖ <close-signal>`. The datestamp is the **same stamp the header carries** (from the `date` call at emit тАФ never re-guessed); the close-signal slot appears only on dispatch validation/close legs. This echo is **deliberate, justified redundancy** тАФ a last-thing-visible orientation aid, distinct from the "a note never regurgitates the entry" rule: it stays **ONE line of key fields**, never a wall; full detail lives in the body above. **The return instruction MUST be command-invoking, not vague (acp-ajudd#43, generalized):** it explicitly tells the receiver to *run `/session:handoff`* to reply with a handoff **block** тАФ never to free-form the report. Running the command is what re-emits this shape; a vague "report back on done" is what let returns come back as loose prose. So a **dispatchтЖТcode work order** footer reads, e.g.:
  ```
   END HANDOFF ┬╖ [2026-07-13 14:32 @ajudd] ┬╖ dispatch (acp-dispatch) тФАтФАтЦ╢ coding ┬╖ #83 ┬╖ Action: PICK UP #83 ┬╖ on IMPLEMENTED-DEPLOYED (or a stop-reason), run /session:handoff to reply with a handoff block back to dispatch тАФ do not free-form the report
  ```
  (Other directions keep the generic `reply to <from-role> on done` return-instruction slot тАФ only notes whose return leg must come back *as a block* need the explicit command invocation.)

**Rules (all required):**
- **Fenced code block is mandatory.** The fence is what gives the Claude UI its one-click copy button and copies the exact raw text тАФ that one-gesture, exact-fidelity copy is the entire point. Never emit a handoff as loose prose.
- **Heavier outer fence when the body has its own fences.** If the body contains ``` fences (bash snippets, JSON, nested blocks), wrap the whole handoff in a **four-backtick** (` ```` `) or `~~~~` fence so the inner triple-backticks survive intact. (This document does exactly that тАФ note the four-backtick wrapper above.)
- **Self-contained body.** Restate all context the receiver needs; never reference "what we decided above" or anything only visible in the originating conversation тАФ the receiving session cannot see it.
- **Titled header + END footer.** The `тХРтХРтХР <FROM-ROLE> тФАтФАтЦ╢ <TO-ROLE> HANDOFF тХРтХРтХР` title tells the receiving Claude what the block is, which role it came from, and **which role it is bound for** (the routing instruction тАФ acp-ajudd#69); the `тХРтХРтХР END HANDOFF тХРтХРтХР` marker tells it where the handoff stops (so trailing chat isn't misread as part of the task).
- **Rule-separated header / body / footer + provenance header.** The `тФАтФАтФА` rules separate the provenance header from the body and the body from the footer (vertical breathing room); the header carries the provenance line + `Re` + `Action`/`State` + `Slug`/`Zone` above. Body stays paragraphed for a human skim while remaining self-contained.

### The courier line тАФ ONE fixed human-facing action-line vocabulary (acp-ajudd#81)

The human courier's cues were patched one at a time (the two-ended title #69, dispatch's SAFE-TO-CLOSE, the coding-finish header #80). They are now **one learnable standard**: every cue aimed at the human тАФ every "carry this / close this / don't close this" тАФ is drawn from **exactly this fixed vocabulary**, no bespoke phrasings. Markers are **emoji, for one-glance contrast** (acp-ajudd#83) тАФ the glyph is the whole point of the cue, so it earns the extra scan-ability. This is safe because the courier line is a **different surface** from hook output: it lives in handoff blocks that Claude emits as chat markdown and the human copy-pastes between terminals тАФ **UTF-8 end to end, never printed to hook stdout**. The repo's ASCII-only rule is scoped to **hook printed output** (Windows `python` fallback stdout is cp1252, where a non-ASCII char crashes the hook) and is **untouched** by this тАФ it does not apply to command/skill `.md` bodies:

- **`тЦ╢ PASTE TO <ROLE> тАФ carry the block below to the <role> terminal`** тАФ the routing cue; pairs with the two-ended title (#69) and accompanies every emitted handoff block.
- **`ЁЯУд <ids> тАФ deployed v<x.y.z> (IMPLEMENTED). Carry the block below to dispatch; keep this terminal open until SAFE-TO-CLOSE.`** тАФ the **ship** cue (the acp-ajudd#94 happy path): code shipped and is reporting up, but the session is **not** closed тАФ shipping is not closing. The coding terminal stays open through dispatch's validation (so a `FIX` can ride the same session), and closes only when the explicit close fires below. (Adapt the deploy clause to what actually shipped.)
- **`тЬЕ <ids> тАФ validated & closed. Safe to close this terminal.`** тАФ the **close** cue, printed by `/session:finish` at the end of the all-or-nothing tie-out (acp-ajudd#94). This is the "safe to close this terminal" moment тАФ it fires only after the close ran, never at ship. (Claude can't restart itself; this cue hands the terminal back to the human to close.)
- **`тЪа STOPPED <ids> тАФ needs input. Carry the block below to <role>; do NOT close this terminal.`** тАФ the escape hatch (#80 stop case: BLOCKED-QUESTION / FOUND-ISSUE / REQUIREMENTS-CHANGE / disagreement).
- **`тП╕ <ids> тАФ <what to look at>`** тАФ dispatch's hold signal.
- **The dispatch close-signal maps IN, it is not a new shape:** `SAFE-TO-CLOSE` is dispatch's post-validation signal тАФ voiced as an `тЬЕ <ids> SAFE-TO-CLOSE тАФ validated; run /session:finish to close.` line (its note-form is the `Action: CLOSE` order, acp-ajudd#94); `HOLD` is voiced as the `тП╕` line. The bare tokens still fill the footer recap's close-signal slot inside the block.

**Placement rule: the action line LEADS тАФ exactly ONE bold line, printed ABOVE the block** (outside the fence), never buried inside the block, below it, or in a summary. One line, not a stack of cues.

**Three cues fire at three points in the human relay (acp-ajudd#80, un-bundled by #94)** тАФ since ship and close are now separate acts, so are their cues: **(1)** at the **build-end ship**, the coding session prints the **`ЁЯУд` / `тЪа STOPPED` header** above its return block тАФ carry the block to dispatch, keep the terminal open; **(2)** **later**, dispatch validates the tree and shows the **`SAFE-TO-CLOSE` / `HOLD`** signal (`Action: CLOSE` as a note); **(3)** back in the still-open coding terminal, `/session:finish` runs the close and prints the **`тЬЕ` safe-to-close** cue. The old #80 model printed a single `тЬЕ close-this-terminal` header at build-end (ship and close were one); #94 splits it тАФ `ЁЯУд` at ship (do **not** close yet), `тЬЕ` at close (now safe). All are needed; none replaces the others.

The emitters (`commands/finish.md`, `commands/dispatch.md`, `commands/handoff.md`) draw their cues **from this vocabulary** тАФ this section is the single source of truth for the wording.

### The close-safety cue тАФ one glyph vocabulary, two always-on mechanisms (acp-ajudd#97)

The courier line above fires at a handoff/ship/close **moment**. This is its **persistent** companion: a per-window, always-on answer to *"can I close this terminal тАФ and in what sense?"* for a human round-robining 3тАУ4 role windows. It protects **unsaved work**; the decision is **pause vs. done-for-good**, per window тАФ so at a glance you know which window to save, which to close, which needs you. ("Busy/working" is deliberately **not** a state тАФ the harness already indicates that.)

**The glyph vocabulary тАФ LOCKED. This one table is the single source of truth; both mechanisms below draw from it тАФ no divergent glyphs anywhere:**

| Glyph | State | Meaning | Close? |
|-------|-------|---------|--------|
| ЁЯЫС | Unsaved тАФ save first | changed work not yet persisted; closing loses it | NO тАФ commit/checkpoint first |
| тЪая╕П | Needs you | question / decision / can't-proceed pending (absorbs "blocked") | no тАФ respond |
| тЬЕ | Saved тАФ safe to close, resume later | persisted (post commit/checkpoint); nothing unsaved; work continues later | yes тАФ pausable |
| ЁЯПЖ | Finished тАФ done & validated | complete + validated + archived; won't reopen barring QA (still editable if truly needed тАФ hence trophy, not a lock) | yes тАФ for good |
| ЁЯУЛ | Carry this | a handoff block is ready to paste into another terminal | carry it first |

The core three are the traffic light: ЁЯЫС red (stop/don't close) ┬╖ тЪая╕П yellow (caution/attention) ┬╖ тЬЕ green (go/clear). **"Idle" is not a separate state тАФ "nothing to save" IS тЬЕ.**

**Two complementary mechanisms тАФ they own different glyphs, so they don't collide:**
- **B тАФ the `statusLine` script (`~/.claude/statusline-command.sh`)** owns the **close-safety light (ЁЯЫС / тЬЕ / ЁЯПЖ)**, derived deterministically from git working-tree state + the active-session `status:`: dirty tree тЖТ ЁЯЫС; clean tree тЖТ тЬЕ; session `completed` тЖТ ЁЯПЖ. Persistent, ANSI-colored (red/green/gold), leading the status line. This is the reliable, safety-critical half. It reads the slug's active-session pointer (`_active` тЖТ `<name>.md`); a window with **no active session file** shows **no save-state** тАФ which is exactly the dispatch carve-out, and also why the sessionless save-states below are carried by A instead.
- **A тАФ a Claude-appended reply footer** owns the **attention states (тЪая╕П needs-you, ЁЯУЛ carry-this)** тАФ only Claude knows it just asked a question/decision or just emitted a handoff block. **End each reply with the footer cue** drawn from the table above. This is the **always-on generalization of the courier line** (┬з above): the courier line fires once, at a handoff moment; this footer carries the window's live state on **every** reply. In a **coding** window B already shows the save-state, so A's footer adds only тЪая╕П / ЁЯУЛ when they apply; in a **sessionless** window (planning / capture) where B is dark, A's footer also carries ЁЯЫС / тЬЕ for that window's save target.

**Role тЖТ save target тАФ same look/feel; only the target behind ЁЯЫС/тЬЕ differs:**
- **coding** тЖТ session file + git (commit / checkpoint / finish). Carried by **B** (git + `status:`).
- **planning (refine)** тЖТ the inbox `work` item тАФ scoped but not yet written = ЁЯЫС; written = тЬЕ. Sessionless тЖТ carried by **A**.
- **capture** тЖТ the capture entry тАФ idea sniffed but not banked = ЁЯЫС; written = тЬЕ (lighter; one-liners). Sessionless тЖТ carried by **A**.
- **dispatch** тЖТ **carve-out:** read-only + notes-only, nothing to persist тЖТ the save-states (ЁЯЫС / тЬЕ / ЁЯПЖ) **never light**; only тЪая╕П / ЁЯУЛ ever apply.

Where the two overlap (finish), acp-ajudd#94's consistent `status:` flip keeps B's ЁЯПЖ and A's finish cue in agreement. **Fast-follow (out of scope here):** Claude-**elevation** of ЁЯЫС when it knows an important change is unsaved though git looks clean (an unwritten decision / memory) тАФ v1 is git-derived save-states (B) + Claude-emitted attention states (A); the smart-elevation overlay lands later.

### Receiving side тАФ verify the target before acting (acp-ajudd#69)

The whole model runs on **human copy-paste between terminals**, so a mis-paste тАФ a note dropped into the wrong terminal тАФ is inevitable, and a receiver that **blindly acts** on it can edit or commit in the **wrong repo** (a real 2026-07-13 incident: a dispatch note meant for one repo was pasted into another terminal, which just did the work). The note already **declares its target** in the header (`Slug:` + the `<to-role>` in the title / `тФАтФАтЦ╢` line); the fix is that the receiver must **check that declaration before doing ANY of the note's work.** This is the **receiver-side** guard that pairs with the sender-side two-ended title above.

**The moment a session detects a pasted handoff block, before acting on it, run this check (instruction-level, no hook тАФ acp-ajudd#1):**

- **PRIMARY тАФ Slug (hard check, ALWAYS runs).** Compute the receiver's own slug from `pwd` (basename of the git toplevel тАФ ┬з Path Resolution) and compare to the note's `Slug:` field. **Mismatch тЖТ STOP, do not act.** This is load-bearing because it works whether the terminal is **fresh** (no role yet) or **established** тАФ a terminal is always running in *some* repo, so a wrong-repo paste is caught regardless of role.
- **SECONDARY тАФ role (conditional).** Only when the receiver **already has an established role** (an active coding session, or a declared `dispatch` / `refine` context): if the note's `<to-role>` contradicts it тАФ e.g. a `coding` work order pasted into an active `dispatch` context тАФ **flag it.** A **fresh terminal with no role SKIPS this** тАФ the note is legitimately *assigning* the role (a `dispatch тФАтФАтЦ╢ coding` order pasted into a brand-new terminal is how that terminal becomes a coding session), so there is nothing to contradict. (This is why fresh terminals gate on **Slug, not role.**)
- **TERTIARY тАФ freshness (optional).** If the note's date is conspicuously old, mention it. Nice-to-have.

**On mismatch тЖТ STOP and flag, never act.** Two message shapes:
- **Established role:** `This note targets <slug> as <to-role>; this terminal is <my-slug> as <my-role>. Not acting - did you mean to paste this here?`
- **Fresh terminal (Slug-only):** `This note targets <slug>; this terminal is in <my-slug>. Not acting - did you mean to paste this here?`

**Every role's flow runs this on detecting a handoff block** тАФ a fresh `code` terminal receiving a work order, a `dispatch` context receiving a return note, a `refine` context receiving an escalation. The two-ended title (sender declares the destination) plus this check (receiver verifies the note is for it) are the **two guards on the one human-relay mispaste risk.**

**The note carries the run; the `work` entry carries the spec тАФ a note never regurgitates the entry.** These are distinct artifacts with distinct jobs:
- The **`work` entry** is the *spec* тАФ scope, acceptance criteria, file list, Done-whens. `refine` authors it; `code` consumes it. It lives in `_inbox/` (one file per item).
- The **handoff note** is the *run* тАФ what to pick up, how to implement, watch-fors, when/how to report back. It **never restates the entry's spec** (the coding session reads the entry itself). A dispatchтЖТcode work order is essentially **`code #X` plus process instructions** тАФ equivalent to a bare `code #X` except that dispatch is coordinating and can attach watch-fors and a report-back protocol.

**Notes are SLEEK POINTERS тАФ the "no spec in the note" hard checklist (acp-ajudd#94).** The rule above is easy to state and easy to violate тАФ a work order bloats into a spec dump the moment it feels helpful to "include the details." It does not: the entry *is* the details, and the coding session reads it. Before emitting a `dispatch тФАтФАтЦ╢ coding` (or `planning тФАтФАтЦ╢ dispatch`) note, check that it does **NOT** contain тАФ copied or paraphrased from the entry тАФ any of:
  - **Done-whens / acceptance criteria** (the coding session reads them off the entry it picks up),
  - **the file list / scope / changes** (same),
  - **the "why" / background prose** the entry already carries.
  A clean work order collapses to roughly *"pick up #X, go, finish when validated"* + the process instructions only dispatch knows (the ship-by-default / don't-self-close + report-back protocol, ┬з The dispatchтЖФcode loop). It **grows past that ONLY for genuine cross-item coordination** a single entry can't hold тАФ sequencing across a bundle, a `depends-on` ordering, or a watch-for that is **truly not in any entry** (e.g. "you're editing the same files #95/#96 will touch тАФ write cleanly"). If a note is repeating what the entry already says, delete that from the note. The dispatch handoff at the head of this very work order is the model: a pointer to the entry + one watch-for, not a re-spec.

**Control lives in dispatch тАФ resolved by who is driving:**
- **Orchestrated** (dispatch sent a note): the **note dictates** the report-back protocol and watch-fors. Control is in dispatch; nothing hardcoded on the coding side fights it.
- **Solo** (no dispatcher тАФ a bare `code #X`, *or* a direct `planning тФАтФАтЦ╢ coding` handoff): the documented default applies тАФ run through, finish, done (the solo carve-out тАФ ┬з The dispatchтЖФcode loop). A bare `code #X` has no one to report to; a **planning-fed** solo session sends a **courtesy** report back to planning, *outside* the strict hub (acp-ajudd#75) тАФ never a `coding тФАтФАтЦ╢ dispatch` relay.
So report-back is **carried by the note**, with a sane solo default.

## The dispatchтЖФcode loop тАФ deploy-then-validate (handed-off work) (acp-ajudd#57, revises #44)

┬з The three roles defines refine / dispatch / code; ┬з Cross-Session Paste Handoff defines the role-aware block that moves work between them. This section is the **protocol that ties them together** тАФ the round-trip **dispatch** and the **coding** session it hands off to run when work is dispatched. It **revises acp-ajudd#44**, which gated the deploy on a planning greenlight (build тЖТ HOLD тЖТ validate тЖТ greenlight тЖТ finish). The new model is **deploy-then-validate**: code ships by default and dispatch confirms afterward. The role model and the block format live in their own sections; this one is layered on top and does not restate either.

**Ship and close are UN-BUNDLED тАФ the keystone (acp-ajudd#94, revises acp-ajudd#57; and #44's history).** #57 said the handed-off session "FINALIZES by default." That bundled two distinct acts тАФ **shipping** (the deploy: bump тЖТ push тЖТ reinstall) and **declaring done** (the close: flip the record to `completed` + deactivate `_active`) тАФ into one automatic step, and the quiet half silently drifted: two shipped sessions (acp-ajudd#77/#78) were found still `active` days after they deployed because the close was *approximated* (a manual commit + a prose edit) instead of run as the full command. #94 splits them:
- **Deploy stays automatic тАФ code's authority.** On the happy path the coding session ships on its own (that part always worked).
- **Close is EXPLICIT and is the ONLY thing that declares "done" тАФ fired only on the validated signal.** A coding session **never** flips its own status to `completed` as a side-effect of shipping. The close is `/session:finish` (the full, all-or-nothing tie-out тАФ `commands/finish.md`), triggered by (a) the human running it on a `SAFE-TO-CLOSE` signal, or (b) a dispatch/planning handoff note carrying `Action: CLOSE`. "Completed" then means **validated + tied out**, by construction тАФ never a session that merely shipped.

**Scope тАФ inbox zones only, handed-off work only.** This loop lives in the dispatch model (┬з The three roles) and applies **only** in the plugin / personal zones and **only** when dispatch hands scoped work to a *fresh* coding session. **Work repos never run it** (no local inbox, no dispatch role тАФ the plain Jira flow). A **solo coding session with no dispatcher just finishes normally** (`/session:finish`) and no round-trip is imposed тАФ see the **solo carve-out** below for the two solo shapes (truly unpaired vs planning-fed). Do **not** impose the loop on unpaired or work-repo sessions.

**The loop тАФ five legs (deploy-then-validate):**
1. **Dispatch hands off** scoped work via `/session:handoff` (a `PICK UP #X` work order), each entry carrying explicit **Done-whens** тАФ the checkable acceptance criteria on the `work` entry it came from (the work layer = requirements + acceptance criteria тАФ see ┬з State-Exclusivity and `references/inbox-convention.md`). The Done-whens **are** the validation contract: code is built to satisfy them, and dispatch later validates against them. The note carries the *run*; the `work` entry carries the *spec* (┬з Cross-Session Paste Handoff).
2. **Code implements.** The **only** reasons to stop and hand a note back (the **escape hatch**) are: a **question**, something **unclear**, **disagreement** with a decision, or a **found problem** тАФ returned as a `State: BLOCKED-QUESTION` / `FOUND-ISSUE` / `REQUIREMENTS-CHANGE` note. Otherwise it does **not** stop.
3. **Happy path тАФ code self-verifies, SHIPS by default, reports `IMPLEMENTED-DEPLOYED`, and does NOT close (acp-ajudd#94).** The coding session self-verifies against the Done-whens and **deploys on its own authority** (bump + push + reinstall тАФ ┬з Development Lifecycle). **No HOLD.** It then, because it was **orchestrated** тАФ detected by having received the leg-1 handoff note **from dispatch** (acp-ajudd#74, Pillar 4, sharpened by #75 тАФ orchestration is detected from a *dispatch* note, not declared) тАФ returns a `State: IMPLEMENTED-DEPLOYED` handoff block to dispatch (command-invoked so it comes back *as a block*, acp-ajudd#43). **What it does NOT do: run `/session:finish`, or flip its own status to `completed`.** The session stays **active** тАФ shipping is not closing (acp-ajudd#94). The close waits for the validated signal (leg 4тА▓). A **solo** session (no note ever received) skips the return тАФ there is no one to report to.
4. **Dispatch confirms post-hoc, and the loop TERMINATES HERE (acp-ajudd#84).** On the return note, dispatch validates the **actual working tree** against the Done-whens тАФ reading the diff / the files code produced тАФ **NOT by rubber-stamping the report.** This is **NOT a gate**: the deploy already happened. Dispatch then shows the human a `SAFE-TO-CLOSE` / `HOLD` close-signal, voiced as a courier line (┬з Cross-Session Paste Handoff тЖТ The courier line), and **pulls the next item itself.** On a clean validation the happy-path loop **ends at dispatch** тАФ there is **no `dispatch тФАтФАтЦ╢ planning` completion report** (┬з The three roles тАФ the happy-path loop terminates at dispatch); planning is left in flow and hears back only on a genuine escalation.
   - **4тА▓. The close is explicit (acp-ajudd#94).** `SAFE-TO-CLOSE` is not just a display тАФ it is the trigger to *close*. The still-open coding terminal runs `/session:finish` (the full all-or-nothing tie-out) either because the human saw `SAFE-TO-CLOSE` and ran it, or because dispatch sent an `Action: CLOSE` note ("validated тАФ run `/session:finish`") the human pasted in. Only `/session:finish` flips the record to `completed`; a session that merely shipped (leg 3) is never `completed`. Because the terminal stays open through validation, a leg-5 `FIX` can ride the same session with no fresh start.
5. **If the post-hoc look finds something off** тЖТ dispatch hands back a `FIX` note тЖТ code fixes and runs **one more deployment** (in the still-open session тАФ leg 4тА▓). Both checks still happen; they are sequential and non-blocking. The fix rides in the **coding session** (a re-deploy), **not** by reopening the consumed `work` entry тАФ the entry is frozen at conversion, and anything genuinely new becomes a fresh `work` entry (┬з The three roles ┬з Inbox write-authority, acp-ajudd#59).

**Rationale (the "why").** Don't freeze code's progress waiting on validation in the common case; the rare miss costs exactly **one extra deploy** тАФ cheap, and two independent checks still occur. Token economy: refine/dispatch stay long/lean, code stays fresh/ephemeral (┬з The three roles). Decoupling implement-progress from scope-progress is the whole point тАФ being mid-refinement never stalls the dispatchтЖФcode loop. **And separating ship from close (acp-ajudd#94)** means "shipped" can never masquerade as "done": the deploy is loud and automatic, the close is a deliberate, atomic, validated act.

**Three load-bearing disciplines** (everything else is mechanics):
- **Code self-verifies and SHIPS тАФ no HOLD (legs 2-3).** A handed-off coding session deploys on its own authority in the happy path; it stops **only** for the escape-hatch reasons (question / unclear / disagreement / found problem). This is the reversal of acp-ajudd#44's greenlight gate тАФ shipping is no longer greenlight-gated.
- **Code ships but NEVER self-closes (leg 3, acp-ajudd#94).** The happy path ends at `IMPLEMENTED-DEPLOYED` with the session still `active`; the coding session does not run `/session:finish` and does not set `completed`. Only the explicit close (leg 4тА▓) declares done.
- **Dispatch VALIDATES the working tree, not the report (leg 4) тАФ post-hoc, non-gating.** Confirmation is earned against the actual diff measured against the Done-whens тАФ never against code's self-report. Independent validation is the payoff of keeping dispatch and code as **separate roles** (┬з The three roles): a validator that did not write the code confirms the build. Only the **timing** moved (after the deploy, not before); the discipline itself is unchanged from #44.

**Solo carve-out тАФ a direct planning тФАтФАтЦ╢ coding handoff is a SOLO session, not a hub chain (acp-ajudd#75).** Dispatch-always is the norm (┬з The three roles), but a planning context sometimes hands work **straight** to a fresh coding session, bypassing dispatch. When that happens the coding session is **solo, not orchestrated** (Pillar 4 тАФ "orchestrated" means a note *from dispatch*): it self-verifies against the Done-whens and **finalizes on its own authority**, exactly like a bare `code #X`. Two solo shapes exist, differing only in the report-back:
- **Truly unpaired** (bare `code #X`, no note ever): finishes normally, reports to no one.
- **Planning-fed** (a `planning тФАтФАтЦ╢ coding` note started it): because a human planner is waiting, the finished session sends a **courtesy** handoff note back to planning. That note is explicitly **outside the strict hub** тАФ a solo report, *not* a `coding тФАтФАтЦ╢ dispatch тФАтФАтЦ╢ planning` two-hop relay тАФ so it does **not** constitute the direct planningтЖФcoding hub edge #74 forbids (that ban is on *hub* edges). The escape hatch (question / unclear / disagreement / found-problem) still applies even solo, handed straight back to planning.

**Load-bearing cost of any bypass:** **no independent validator runs** тАФ the same session that wrote the code is the only one that checks it, forfeiting dispatch's core value (a checker who did not write the code, leg 4). A direct handoff is therefore safe **only for doc-only work with crisp Done-whens**; behavior-changing work should route through the dispatch hub. This is why the norm is dispatch-always and the bypass is a deliberately narrow carve-out тАФ **not** a sanctioned new hub edge (the "add an exception edge" option was rejected).

## HALT тАФ standing down dispatched work mid-flight (acp-ajudd#67)

**Two words in the handoff field set (┬з Cross-Session Paste Handoff):** `Action: HALT` (outbound stand-down тАФ stop the dispatched work cleanly mid-flight) and `State: HALTED` (return leg тАФ stopped, and in what state).

**Full detail moved to `references/halt.md`** (acp-ajudd#78). Load it when dispatching or receiving a HALT. In short: HALT is the *other* exit from the dispatchтЖФcode loop (not completion, not the escape hatch) тАФ *stop, don't resume this run*; the clean-stop procedure is **no publish, no commit/deploy, preserve the WIP, return a `State: HALTED` block**; and **halted work re-enters as a NEW entry citing the halted one** (distinct from consumed = frozen), never by reopening тАФ as acp-ajudd#65 did for the paused #60.
